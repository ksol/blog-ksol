Because of Ember's support for bindings and observers, accessing and setting properties on objects has to be done using the get and set methods, respectively. Most of the times, you will use object.get('myproperty') or something close to it. But in some cases, when reading code or libraries source, you may stumble upon Ember.get(object, 'myproperty') instead. What's the difference and when should you use one or the other?

When reading the source, you will see that #get is defined in the Observable mixin, and all the method does is call Ember.get, so that object.get('myproperty') will call Ember.get(object, 'myproperty'). Therefore, both will always be equal, and the same goes for Ember.set. But there is at least two scenarios where you may want to use Ember.get instead of calling #get:

Plain old javascript object

You cannot call #get on objects that does not extend Ember.Object (or Ember.Observable). But Ember.get works on pure javascript objects, with the same semantics as #get:

Ember.set works similarly, and allows you to set properties in pure javascript objects as if they were extending Ember.Object.

Null objects

When called on null, using Ember.get instead of #get will return undefined instead of raising a TypeError. If you know and expect that your object might be null, this is probably what you want.

Be careful though: calling Ember.get with undefined will still raise an exception:

Ember.set does not work with null objects, and this is probably why it is not as used as Ember.get.

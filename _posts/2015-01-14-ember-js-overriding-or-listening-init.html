Brian Cardarella wrote some time ago that you should not override the method init, but rather listen to the event of the same name. His article makes lots of sense, but sadly the init event is still little known. So I thought I could update the API docs and make it a little less hidden.

This triggered an interesting discussion on the differences between overriding the method or listening to the event.

Regarding the init method :

  • its content will not trigger change events;
  • you can control the order of execution inside the method;
  • you can refine what _super does.

As for the event init:

  • handlers will trigger change events;
  • you cannot be assured of the call order if there is more than one event handler on init;
  • you cannot (or not easily) refine previously attached handlers.

I still think the event is often more useful than overriding, but the differences listed above should be known. When writing a library, for instance, you probably should override the method so the users can feel free to refine what init does if they need to.

The Ember team will discuss this issue and we can expect some clarification on the role of the event in the future. When they do, I'll let you know where we stand on this matter.
